# NeurSC 训练文档（GPU 实操：900 点图全流程）

本文给出在 **GPU** 上从零跑通一次完整实验的命令：**生成 900 点地形 → 训练 → 推理测试 → 与非学习基线对比**。
命令在**项目根目录**执行（源码 `.py` 都在根目录，路径默认相对项目根解析）。

> 三个脚本相互独立、按需运行：`generate_terrain.py`（造地形）/ `main.py`（训练）/ `baseline.py`（基线）。
> 训练只读 `--off_file` 指向的 `.off`，不会自动触发地形生成。

---

## 0. 云服务器从零开始（新手必读，一次性准备）

> 目标：拿到一台带 GPU 的云服务器后，照这一节把代码跑起来。逐步做，不要跳。

### 0.1 把代码放到服务器上

二选一：
- **git 克隆**（项目在远程仓库时）：`git clone <仓库地址>` 然后 `cd <项目目录>`；
- **直接上传**：用服务器面板的"上传"或 `scp`/`FileZilla` 把整个项目文件夹传上去，再 `cd` 进去。

确认你在**项目根目录**（能看到 `main.py`、`generate_terrain.py`、`data/`、`outputs/` 等）：
```bash
ls
```
> 下面所有命令都在这个根目录里执行。

### 0.2 配环境（关键，别把服务器自带的 torch 装坏）

云 GPU 服务器（AutoDL 等）**通常已预装配好 CUDA 的 PyTorch**。先确认它能用：
```bash
python -c "import torch; print(torch.__version__, torch.cuda.is_available())"
```
- 打印出版本号且第二项是 `True` → torch 没问题，**不要重装 torch**，只补装其余依赖：
  ```bash
  pip install numpy tqdm torch-geometric
  ```
- 如果上面那行 **报错 “No module named torch”**（说明服务器没预装），才整体装：
  ```bash
  pip install -r requirements.txt
  ```

> ⚠️ 不要在已经有 GPU 版 torch 的服务器上直接 `pip install -r requirements.txt`——它可能把 torch
> 覆盖成 CPU 版或 CUDA 不匹配的版本，导致 `--device cuda` 失效。
>
> 若 `import torch_geometric` 报错，多半是版本不匹配，按其官网说明选与当前 torch/CUDA 对应的安装方式；
> 实在装不上可先用 `--device cpu` 跑通流程（慢，但能验证代码无误）。

### 0.3 冒烟测试（务必先跑，30 秒级）

正式训练前，先用极小配置确认整条链路通（生成 → 训练一轮 → 出结果）：

`outputs/models/` 下生成了 `.pt` 文件。通过后再进入第 1 节的正式流程。

> 若冒烟测试报错，**别自己硬改**，把下面信息发给负责人（见 0.4）。

### 0.4 出错了要带回的信息（照抄给负责人）

## 1. 生成 900 点地形

```bash
python generate_terrain.py --grid 30
```

- `--grid 30` → 30×30 = **900 个顶点**，输出到 `data/generated/terrain_grid_30x30_900v.off`。
- 打印里会显示顶点数、面数、高度起伏 `z_range / z_std`，确认地形有起伏（不是平地）。
- 可选：`--relief 0.5` 让地形更陡（图最短路更偏离直线，更能体现方法价值）；`--seed` 换地形。

> 这一步纯 CPU、秒级完成。生成一次即可，后续反复训练都复用这张图。

## 2. 在 GPU 上训练

```bash
python main.py \
  --off_file generated/terrain_grid_30x30_900v.off \
  --max_depth 3 --capacity 32 \
  --loss_type log_l1 \
  --distance_samples 100000 \
  --num_epoch 30 \
  --batch_size 32 \
  --device cuda 2>&1 | tee outputs/logs/EP_low_d3_c32_100ksamples.log
```

> **日志归档**：命令末尾的 `2>&1 | tee outputs/logs/<描述性名字>.log` 会把训练输出**同时打印到屏幕并存档**到
> `outputs/logs/`。文件名建议带上关键参数（图/depth/capacity/样本数/epoch），便于日后按参数/版本对比。
> 例如 `EP_low_d3_c32_100ksamples.log`。这个目录专门存不同参数、不同版本的训练日志。

参数说明（与本次实验相关的重点）：

| 参数 | 取值 | 含义 |
|---|---|---|
| `--off_file` | `generated/...` | 默认相对 `data/` 解析，即 `data/generated/terrain_grid_30x30_900v.off` |
| `--max_depth` / `--capacity` | `3` / `32` | 四叉树分区参数（**推理/基线必须与此一致**） |
| `--distance_samples` | `100000` | 采样 100000 个 `(s,t)` 监督对（约占 900 点全部对 ~40 万的 25%）；`<=0` 表示用全部唯一对 |
| `--num_epoch` | `30` | 最大训练轮数（含早停） |
| `--batch_size` | `32` | mini-batch 大小，提升 GPU 利用率；显存不够就调小（16/8） |
| `--device` | `cuda` | 用 GPU 训练 |

**启动顺序与耗时**（日志会打印各阶段时间）：
1. 首次会先做四叉树分区 + 对每个高速节点跑 Dijkstra（**纯 CPU**），完成后打印
   `preprocess(分区+高速/缓存) done in X.XXs` 并把结果存到 `outputs/cache/`，**下次相同图+参数直接秒读**。
2. 之后逐轮训练，每轮打印 `train_mae / val_mae / train_rmse / val_rmse / time=XX.XXs`，按 `val_mae` 早停。
3. 结束打印 `test_mae / test_rmse / test_relative_error` 与
   `timing: preprocess=...s, train=...s, best_epoch=...`。

**产物**（`<run>` = `terrain_grid_30x30_900v_distance_<时间戳>`）：

| 路径 | 内容 |
|---|---|
| `outputs/models/<run>.pt` | 训练好的模型权重 |
| `outputs/results/<run>.txt` | 测试指标 + 耗时（best_val_mae / best_epoch / test_* / preprocess_seconds / train_seconds） |
| `outputs/results/<run>_test_pairs.csv` | 本次 test 的 `(s,t,真实距离)`，供基线对齐 |
| `outputs/params/<run>.txt` | 本次运行的全部参数快照 |
| `outputs/cache/<key>*.{pt,csv}` | 分区+高速缓存与审查文件 |

> 记下命令打印的 `<run>` 名（或去 `outputs/models/` 看），第 3、4 步要用。

## 3. 推理测试（用训练好的模型预测某两点距离）

参数（`--max_depth/--capacity/--in_feat/--inner_mode`）**必须与训练一致**，否则分区/模型对不上。
顶点编号范围 `0 ~ 899`。

```bash
python infer_distance.py \
  --model_path outputs/models/<run>.pt \
  --off_file generated/terrain_grid_30x30_900v.off \
  --max_depth 3 --capacity 32 \
  --s 10 --t 800 \
  --device cuda
```

输出形如 `predicted_distance(10->800) = 31.4827`。换 `--s / --t` 可查任意两点。

> 推理会命中第 2 步生成的 `outputs/cache/` 缓存，启动很快。

## 4. 与非学习基线对比性能

基线在**训练导出的同一批 test 点对**上评估（绝对同口径），纯 CPU、无需 GPU：

```bash
python baseline.py \
  --off_file generated/terrain_grid_30x30_900v.off \
  --test_pairs_file outputs/results/<run>_test_pairs.csv \
  --out_file outputs/results/<run>_baseline.txt
```

输出三个朴素基线的 `mae / rmse / relative_error`：

- `mean_constant`：永远预测 test 真值均值（最优常数基线）；
- `euclidean_2d`：忽略高度的平面直线距离；
- `euclidean_3d`：3D 直线距离。

**怎么判断模型有价值**：把第 2 步训练打印的 `test_relative_error` 和这里三个基线的 `relative_error` 并排比。
模型的相对误差**低于**这些基线，就说明它在朴素方法之上确有提升。

## 6. 关于耗时记录

训练日志与 `outputs/results/<run>.txt` 都会记录两段时间，便于判断瓶颈在哪：

- `preprocess_seconds`：四叉树分区 + 高速 Dijkstra 预处理（纯 CPU；命中缓存时会很小）。
- `train_seconds`：训练总时长（GPU 部分）。
- 每轮日志的 `time=XX.XXs`：单轮耗时，用于观察 batch_size / 设备对速度的影响。

> 若发现 `preprocess_seconds` 占大头，说明瓶颈在 CPU 预处理，靠缓存（第二次跑秒读）即可缓解；
> 若 `train_seconds` 占大头，可调大 `--batch_size` 提升 GPU 吞吐（注意显存）。

## 7. 常见问题

- **显存不足（CUDA out of memory）**：调小 `--batch_size`（32 → 16 → 8）。
- **启动卡在分区/Dijkstra**：纯 CPU 预处理，首次较慢；完成后已缓存，第二次起秒读。想强制重算加 `--no_cache`。
- **改了地形或四叉树参数后想重训**：缓存键含 `.off` 内容指纹 + 参数，会自动用新缓存，不会读到旧的。
- **推理/基线结果对不上**：检查 `--off_file` 与 `--max_depth/--capacity/--in_feat/--inner_mode` 是否和训练完全一致。
- **想跑别的规模**：`--grid 10`→100 点、`--grid 20`→400 点、`--grid 40`→1600 点；文件名 `terrain_grid_NxN_<V>v.off` 与顶点编号上限同步替换。
